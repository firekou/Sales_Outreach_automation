> 版本：v1.0 | 日期：2026-06-09 | Owner：Frank Kao
> 階段：Phase 1（半人工驗證），預計執行 4 週，挑選 20 家黃金窗口公司
> 前置：Phase 0 合規閘門通過（104 取得方式定案）
> 配套：`linkedin-outreach-v2-intent-sourcing.md`（方法論）、`linkedin-outreach-v2-ta-matrix.md`（TA 矩陣）

## 0. Phase 1 的唯一目標

> **驗證假設：在意圖窗口（T+14~45）、引用 JD 痛點觸達，回覆率是否達到 V1 的 2–3 倍（≥20% vs V1 的 8%）。**

Phase 1 不求規模，求驗證。20 家公司，每家走完整 I2DM 鏈路，記錄數據，2–4 週出結論。
數據成立 → 投資 Phase 2 自動化；數據不成立 → 修正方法論。

## 2. 每週操作流程（SOP）

### Step 1：掃描意圖訊號（每週一，30 分鐘）

**工具：TheirStack 或手動 104**

篩選條件：
- 職缺發布日：過去 7 天
- 關鍵字命中：`LLM / RAG / GPT / Claude / 生成式AI / AIGC / Prompt Engineering`
- 公司規模：5–1000 人（排除個人接案）
- 地區：台灣

目標：每週挑出 5–8 個新候選公司，填入追蹤表（見第 3 節）。

---

### Step 2：意圖分級 + 部門溯源（每家 10 分鐘）

打開 JD，對照以下清單快速判斷：

**Tier 判定**

| 判定條件 | Tier |
|---------|------|
| 職缺為「Head of AI」/ 「AI 負責人」/ 同公司 3+ AI 職缺 | **S** |
| JD 明確點名 GPT / Claude / Gemini / RAG / Agent | **A** |
| 「資料科學家」/ 「數位轉型 PM」/ JD 僅順帶提 AI | **B** |
| 「Python 工程師」/ 泛數位 / 無具體 AI 用例 | C（跳過） |

**加分訊號（疊加）**

| 訊號 | 加分 |
|------|------|
| 同公司 3+ AI 職缺 | +10 |
| 職缺 >60 天或重貼 | +8 |
| JD 明確點名多模型（GPT + Claude + 其他）| +8 |
| 高薪（高於市場均值 20%+）| +5 |

**TA 匹配**（對照 `linkedin-outreach-v2-ta-matrix.md`）

記錄：主要 TA（TA-1 ~ TA-6）、對應痛點句、決策者職稱。

---

### Step 3：計算意圖窗口位置（每家 2 分鐘）

```
今天距 T0（職缺發布日）= _____ 天

0–14 天 → 軟觸達期（先連結，不推銷）
14–45 天 → 🔥 黃金窗口（立即進入全流程）
45–90 天 → 競爭期（仍可觸達但效果較低）
90 天+ → 培育池（標記，等下次訊號升級）
```

Phase 1 優先選「落在黃金窗口（14–45 天）」的公司。

---

### Step 4：決策者解析（每家 10–15 分鐘）

```
1. Google：「公司名 + LinkedIn」找公司頁面
2. LinkedIn 公司頁 → People → 篩選：部門（IT/工程/產品/策略）+ 職級（Director 以上）
3. Sales Navigator（Frank 帳號）：公司名 + 職稱關鍵字搜尋
4. 判斷任期：決策者到職日 < T0（否則他不是核准此 headcount 的人）
5. <50 人公司直接找創辦人 / 執行長
```

記錄：LinkedIn URL、決策者姓名、職稱、到職日、LinkedIn 活躍度（有/無近期動態）。

---

### Step 5：計算 Opportunity Score（自動，填入追蹤表）

```
Opportunity Score =
  Intent Strength (0–40)   = Tier 分數（S:40 / A:30 / B:18）+ 加分訊號
+ Timing Freshness (0–25)  = 黃金窗口:25 / 招募期:15 / 競爭期:12 / 培育:5
+ ICP Fit (0–20)           = 公司規模 + 產業吻合度（參考 V1 ICP 評分維度）
+ Reachability (0–15)      = LinkedIn 活躍:15 / 沉睡:9 / 完全無 LI:3
```

Phase 1 觸達閾值：總分 ≥ 55（WARM-INTENT 以上）。

---

### Step 6：生成個人化訊息（每家 10 分鐘）

**開 Claude，使用以下 Prompt 框架：**

```
你是 AI Token King 的業務代表（[帳號人員姓名]，職稱：[title]）。

目標對象：
- 姓名：[決策者姓名]
- 職稱：[職稱]
- 公司：[公司名稱]
- 公司規模：[規模]
- TA 分類：[TA-1 ~ TA-6]
- 偵測到的意圖：[JD 職缺名稱]
- JD 關鍵段落（直接貼）：
  [貼上 JD 最能反映痛點的 2–3 段]

請生成：
1. 連結請求附言（≤300 字元）：親切、不推銷，好奇對方的 AI 佈局
2. 價值觸達訊息（≤500 字元）：引用 JD 具體痛點，給出一個具體可交付物，附 CTA

訊息風格：[對應 TA 的訊息風格，見 TA 矩陣]
```

記錄：貼入追蹤表 custom1（連結請求）、custom2（價值觸達）欄位。

---

### Step 7：記錄 & 執行觸達（每週三～四）

1. 將本週批次填入「Phase 1 追蹤表」（見第 3 節）
2. 核對 Asana：用 `asana_dedup.py lookup` 確認尚未觸達
3. Frank 在 LinkedIn 手動發出連結請求（含 custom1）
4. 記錄「發出日期」欄位
5. 連結接受後第 3 天，手動發出 custom2

---

### Step 8：週五回顧（15 分鐘）

- 統計本週：掃描數 / 進入追蹤表 / 觸達數 / 連結接受 / 回覆
- 比對 KPI 目標（見第 4 節）
- 標記任何「黃金窗口假設不成立」的案例（回覆了但超出窗口、窗口內但沒回）

## 4. KPI 驗證儀表板（4 週後評估）

### 核心假設驗證

| 指標 | V1 基準 | Phase 1 目標 | 驗證方式 |
|------|---------|-------------|---------|
| Connection 接受率 | ≥25% | **≥40%** | 接受數 / 發出數 |
| 價值觸達回覆率 | ≥8% | **≥20%** | 回覆數 / custom2 發出數 |
| 黃金窗口命中率 | — | ≥60% | 黃金窗口內觸達 / 總觸達 |
| 意圖→Demo 轉化 | ≥3% | **≥10%** | 進入 Demo 討論 / 觸達數 |

### 子假設拆解（供Phase 1 排除法使用）

| 假設 | 驗證切法 |
|------|---------|
| 黃金窗口比非窗口回覆率高 | 比較「距T0 14~45天」vs「其他時段」的回覆率 |
| JD 引用比無引用回覆率高 | 比較有無直接引用 JD 段落的草稿差異（人工拆分測試） |
| Tier S/A 比 B 回覆率高 | 按 Tier 分組統計 |
| <50人公司比大公司更快回 | 按公司規模分組統計 |

## 5. 決策關口：4 週後怎麼判斷

### 繼續推進 Phase 2（全部條件達標）

- [ ] 連結接受率 ≥ 35%
- [ ] 價值觸達回覆率 ≥ 15%
- [ ] 至少 2 家進入 Demo 討論
- [ ] 黃金窗口回覆率明顯高於窗口外（差距 ≥ 10 個百分點）

### 修正方法論（部分指標未達，但有正訊號）

若回覆率低但黃金窗口組好過非窗口組 → **窗口假設成立，問題在訊息或 TA 匹配**
→ 修正：調整訊息框架、重新校準 TA 匹配矩陣，再跑第二批 20 家

### 暫停並重新評估（指標全面不如 V1）

若接受率 < 20% 且回覆率 < 5% → 方法論可能有根本性問題
→ 召集 Frank + Jet + Lauren 評估是否：
  (a) 台灣市場對「意圖觸達」的接受度低
  (b) TA 矩陣 / 訊息框架需大幅修正
  (c) 數據源不足（20 家樣本量太小）

## 8. 注意事項 & 紅線

- **每日發出上限：25 則連結請求（Frank 帳號），不超標**
- **必須 Asana 去重後才發**：用 `asana_dedup.py lookup <url>` 確認
- **訊息不可包含：** 價格數字、「免費試用」、競品比較、過度推銷語氣
- **JD 引用必須精確：** 引用的技術棧、用例必須確實出現在該公司 JD，不可捏造
- **Phase 1 觸達帳號：** 全部用 Frank 帳號（確保數據乾淨，單一變數）
- **資料存放：** 追蹤表 Google Sheets 存於兌心科技 Drive；含個資欄位（決策者資料）僅供業務內部使用，不對外分享
