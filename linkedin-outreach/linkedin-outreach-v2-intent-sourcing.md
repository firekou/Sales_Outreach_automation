## 0. 一句話定位

> **在客戶還沒察覺自己已展露意圖時，從公開徵才訊號溯源到實際決策者，搶在黃金窗口、用對方自己寫的痛點，跨平台精準觸達。**

V1.0 是「在 LinkedIn 上找對的人」。
V2.0 是「在客戶展露意圖的當下，找到對的時機、對的人、對的通道」。

## 1. 核心洞察：為什麼 104 是最強意圖訊號源

當一家公司在 104 貼出 AI 相關職缺，它免費揭露了四件事：

| 揭露維度 | 範例 | 對 Sales 的價值 |
|---------|------|---------------|
| **要做什麼**（用例）| 「建置內部 RAG 知識庫」 | 知道他的專案 = 知道痛點 |
| **用什麼**（技術棧）| 「整合 GPT-4 與 Claude API」 | 多模型 = AI Token King 命中點 |
| **什麼時候**（時程）| 職缺發布日 = 需求啟動點 | 計算黃金觸達窗口 |
| **誰買單**（預算）| 核准 headcount 的主管 | 反推經濟買家 |

**關鍵差距：** 這比客戶填寫「contact us 表單」早 3–6 個月，且資訊密度高 10 倍。
JD 本質上是「客戶在成為客戶之前，用自己的話寫下的需求說明書」。

### 競爭護城河（為何是「別人尚未發現的商機」）

1. 西方意圖數據工具（Bombora、ZoomInfo Intent）**不覆蓋台灣 / 104**
2. 台灣 B2B 銷售仍以關係 / 陌生電話為主，**幾乎無人做訊號驅動（signal-based）**
3. 104 → LinkedIn 的跨平台溯源鏈，**在台灣 AI 工具市場是全新打法**
4. 時機優勢：在 T+14~45 天窗口接觸，**早於客戶主動比價**

## 2. 戰略反轉：V1 vs V2.0

| 維度 | V1.0（現行） | V2.0（意圖溯源） |
|------|------------|----------------|
| 起點 | 職稱 / 產業（他是不是對的人） | 意圖訊號（他是不是現在有需求） |
| 訊號源 | LinkedIn Sales Navigator | 104 徵才 → 反推 LinkedIn |

| 通道 | LinkedIn 單一 | LinkedIn / Email / FB / 官網 多通道瀑布 |

因為台灣大量 SME 老闆 / 傳產決策者根本不在 LinkedIn，意圖才是主軸，通道是手段。

## 4. 意圖訊號分級（Intent Ladder）

依職缺反映的承諾強度分四級：

### Tier S — 戰略級意圖（最高，董事會級承諾）
- 「Head of AI」「AI 負責人」「生成式 AI 團隊主管」「Chief AI Officer」
- **同時貼出多個 AI 職缺**（= 在建團隊）
- → 經濟買家通常是 CEO / 創辦人，預算大、決心強

### Tier A — 強意圖（積極建置中）
- 「LLM 應用工程師」「生成式 AI 工程師」「AI 產品經理」「MLOps 工程師」
- JD 明確提及 GPT / Claude / Gemini / RAG / Agent
- → 經濟買家 CTO / 技術長 / 產品總監

### Tier B — 中度意圖（探索中）
- 「資料科學家」（可能是傳統 ML）「數位轉型 PM」「AI 行銷專員」
- → 需進一步看 JD 判斷 AI 純度

### Tier C — 弱 / 模糊
- 「Python 工程師」JD 順帶提 AI、泛「數位」職缺
- → 低優先，僅放培育池

### 加分訊號（疊加在 Tier 之上）
| 訊號 | 含義 | 加分 |
|------|------|------|
| 同公司多個 AI 職缺 | 擴張中，超熱 | +10 |
| 職缺開 >60 天 / 重貼 | 招不到人 → **更可能需工具/服務替代** | +8 |
| 高薪資帶（高於市場） | 預算充足、決心強 | +5 |
| JD 點名多模型 | 直接命中 AI Token King | +8 |

## 5. 部門溯源與決策者映射

JD 內容反推「誰核准了這個 headcount = 誰是經濟買家」：

| JD 訊號 | 需求源部門 | 經濟買家 | 技術買家 | <50 人時 |
|---------|-----------|---------|---------|---------|
| LLM工程師 / MLOps / AI後端 | RD / 工程 | CTO / 技術長 | CTO | **創辦人** |
| AI 產品經理 / GenAI PM | 產品 | CPO / 產品總監 | CTO+CPO | 創辦人 |
| AI 行銷 / 內容生成 | 行銷 | CMO / 行銷總監 | 行銷總監 | 創辦人 |
| 數位轉型 PM / AI 導入顧問 | CEO office / 策略 | **CEO / 總經理 / COO** | CIO | 創辦人 |
| 資料科學家 | 數據 | 數據長 / CTO | 數據主管 | 創辦人 |
| AI 標註 / 資料處理 | 營運 | 營運主管 | （低價值） | 創辦人 |

**鐵則：公司越小，決策者越往 CEO 集中。** 50 人以下直接找老闆，不繞部門。

## 6. 意圖窗口時序模型（V2.0 獨有）

職缺發布 = T0：

```
T0 ─────── T+14 ─────────── T+45 ─────────── T+90 ──────▶
 │  招募期    │   🔥黃金窗口      │   競爭期        │  轉培育
 │  HR 篩人   │  新人到職、發現   │  自己解決 or    │
 │  軟觸達    │  需要基礎建設     │  已在比價       │
 │ (僅連結,   │  硬價值觸達       │  仍可進但較貴   │
 │  不推銷)   │  (主力火力)       │                 │
```

| 窗口階段 | 距 T0 | 觸達策略 | 訊息重點 |
|---------|-------|---------|---------|
| 招募期 | 0–14 天 | 軟觸達（僅建立連結，不推銷） | 對其 AI 佈局表達好奇 |
| **黃金窗口** | **14–45 天** | **硬價值觸達（主力）** | 直接命中 JD 痛點 + 可交付物 |
| 競爭期 | 45–90 天 | 標準觸達 | 差異化、案例 |
| 培育 | 90 天+ | 移入培育池，季度再觸 | 內容 nurture |

> **執行週期：** 每雙週 / 每月掃描，鎖定「當下落在黃金窗口」的所有公司。

## 7. 決策者解析（瀑布）

```
Step 1  LinkedIn 公司頁 → People → 篩選「推斷部門 + 資深度」
Step 2  Sales Nav：公司名 + 職稱關鍵字搜尋
Step 3  任期驗證：到職日須早於職缺發布日（他才是決定招人的人）
Step 4  多候選打分：資深度 + 部門吻合度 + 任期 → 取最高分
Step 5  小公司捷徑：<50 人直接鎖定創辦人 / 總經理
```

## 8. 多平台觸達瀑布（Reachability Waterfall）

台灣現實：大量老闆 / 傳產決策者不在 LinkedIn。通道策略需分層：

```
評估目標 LinkedIn 狀態
  │
  ├─ 活躍（近 30 天有動態）   → 觸達分 5 → LinkedIn 直球（V1 流程）
  │
  ├─ 有帳號但沉睡             → 觸達分 3 → LinkedIn 連結 + Email 備援
  │
  └─ 完全無 LinkedIn          → 觸達分 1 → 啟動替代通道瀑布：
        ① 公司 Email（猜 pattern：name@company.com.tw + Hunter/Apollo 驗證）
        ② 公司官網「聯絡我們」/「領導團隊」頁
        ③ Facebook（台灣高管活躍地，尤其傳產老闆）
        ④ 產業公會 / 活動名單
        ⑤ 二度人脈暖引薦（LinkedIn 2nd degree / 共同朋友）
        ⑥ 104 徵才聯絡窗口（最後手段：請求轉介用人主管）
```

**觸達預算原則：**
- 高意圖（Tier S/A）+ 低可觸達 → 值得投資多通道（Email + FB + 暖引薦）
- 低意圖（Tier C）+ 沉睡 → 一封 LinkedIn 即止，不浪費資源

## 9. 機會評分模型（Opportunity Score）

每日 / 雙週目標清單的排序依據：

```
Opportunity Score (0–100) =
    Intent Strength   (0–40)   ← 職缺 Tier(S40/A30/B18/C8) + 加分訊號
  + Timing Freshness  (0–25)   ← 黃金窗口 25 / 招募期 15 / 競爭期 12 / 培育 5
  + ICP Fit           (0–20)   ← 沿用 V1 既有評分（公司規模 / 產業）
  + Reachability      (0–15)   ← 觸達分 5/3/1 → 映射 15/9/3
```

| 總分 | 分級 | 行動 |
|------|------|------|
| ≥75 | HOT-INTENT | 黃金窗口內優先觸達，多通道佈局 |
| 55–74 | WARM-INTENT | 標準觸達，主通道 |
| 35–54 | COLD-INTENT | 培育池，等窗口或訊號升級 |
| <35 | DROP | 不觸達 |

## 12. ⚠ 合規閘門（進入工程實作前必過）

**這是 P0 阻斷項，未通過不得建置 104 scanner。** Owner：Jet（風控）+ Alice（產品）+ 法務。

### 12.1 104 服務條款（ToS）
- 104.com.tw ToS 通常**禁止自動化爬取**，違反可能導致帳號封鎖或法律風險
- **合規路徑選項（擇一）：**
  1. **104 官方 eRecruiter API**（已確認存在）：`developers.104.com.tw`，需企業夥伴申請，適合 Phase 3+ 規模化 → **長期首選**
  2. **TheirStack**（立即可用）：195 國含台灣，免費試用 50 家公司 + 200 API Credits，從 JD/GitHub/官網技術堆疊偵測意圖 → **Phase 1 推薦**
  3. **Lightcast（Burning Glass）**：2026/3 起涵蓋台灣在內共 165 國，企業授權，精準度高 → **Phase 2+ 評估**
  4. **半人工掃 104**：人員手動瀏覽記錄 → 慢但零合規風險，Phase 1 備案
  5. **CakeResume**：無公開 API，Apify 上無台灣特化 Actor，僅能手動瀏覽，不建議自動化

> **Phase 0 行動項：** Frank 開 TheirStack 免費試用帳號，測試台灣 AI 意圖公司覆蓋廣度；
> 同步評估 104 eRecruiter API 企業申請可行性（Jet 商務接洽）。

### 12.2 個資法（PDPA）
- 公司徵才資訊 = 法人資訊，**非個資，風險低**
- 決策者個人檔案 = 個資；B2B 觸達有「特定目的」latitude，但須：
  - 來源須為**合法公開的專業檔案**
  - 提供 **opt-out 機制**
  - 不得過度蒐集 / 建檔
- 路由現有 `legal/legal-compliance-notes.md` 由 Alice + Jet 複核

### 12.3 決議要求
工程實作啟動前，須取得：
- [ ] Jet 風控對 104 數據取得方式的書面核准
- [ ] Alice + 法務對 PDPA 觸達合規的確認
- [ ] 104 數據取得方式定案（API / 半人工 / 第三方）
