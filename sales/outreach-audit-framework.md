# AI Token King — 陌生開發查核與迭代框架

**版本：** 1.0  
**日期：** 2026-05-31  
**核心理念：** 數字不代表真相，必須主動查核每一層的真實性  
**Owner：** A-LI-Auditor（查核）× Kid（執行）× Lauren（裁決）

---

## 為什麼需要獨立查核？

```
業務開發的自欺陷阱：

  「我發了 100 封訊息」≠「100 個人看到了」
  「50 個人接受連結」≠「50 個人對你有興趣」
  「10 個人回覆了」≠「10 個人是真實潛在客戶」
  「5 個人說好像有興趣」≠「5 個人會去看 Demo」
  「2 個人看了 Demo」≠「2 個人會成交」

每一層都有「數字膨脹」的傾向，
越往後的步驟，樂觀偏差越嚴重。

A-LI-Auditor 的工作就是在每一層打折，
找到真實數字，然後告訴 Kid 和 Lauren 實情。
```

## 七層查核體系

### Layer 1：Lead 訊號真實性（Lead Signal Authenticity）

**問題：這個「訊號」是真的嗎？**

```
LinkedIn Lead 真實性查核清單：

  Profile 完整度（≥ 3 項才算真實）：
    □ 帳號成立 > 2 年
    □ 連結數 > 100 人
    □ 有過去 3 個月的貼文或互動記錄
    □ 有清楚的工作經歷描述（不只是職稱）
    □ 有照片（非卡通/空白）
    □ 公司名稱可以在 Google 找到真實網站

  公司真實性（≥ 2 項才算真實）：
    □ 公司有官網（非只有 Facebook）
    □ LinkedIn 公司頁有 > 10 位員工標記
    □ 公司有 Google 商業檔案或 104 頁面
    □ 員工規模與 ICP 描述一致

  AI 訊號真實性：
    □ Profile 提到 AI/雲端（非只靠關鍵字推算）
    □ 貼文有實際分享 AI 工具使用經驗
    □ 職缺描述明確提到具體 AI 工具名稱

  A-LI-Auditor 自動評分：
    三項全過 → 訊號可信度 HIGH
    部分通過 → 訊號可信度 MEDIUM（需 Kid 人工確認）
    多項未過 → 訊號可信度 LOW（建議從 Lead Pool 移除）
```

### Layer 2：訊息送達驗證（Delivery Verification）

**問題：訊息真的被看到了嗎？**

```
LinkedIn 連結請求：
  已送出（Sent）≠ 已看到（Viewed）
  
  驗證方法：
    - 連結請求 > 14 天未處理 → 可能帳號不活躍
    - 若對方有最近貼文但不接受 → 主動看到但拒絕（真實拒絕）
    - 若對方沒有近期活動 → 沉睡帳號，不計入有效觸達

LinkedIn InMail：
  送出 → 查看「已讀」或「未讀」狀態
  有已讀 → 確認看到，回覆率計算基數用此數字

Apollo Email：
  發出 → 開信追蹤像素確認開信
  開信 ≠ 閱讀（部分郵件客戶端自動載入圖片）
  
  更可靠的送達指標：
    ✓ 連結點擊（最強確認信號）
    ✓ 回覆（最終確認）
    ✓ 開信 ≥ 3 次（多次開信 = 有在考慮）
    
  退信分類（每週 A-LI-Auditor 自動分析）：
    Hard Bounce（信箱不存在）→ 移除，更新 Apollo 資料
    Soft Bounce（信箱暫時問題）→ 3 天後重試一次
    退信率 > 3% → 立刻暫停序列，通知 Kid
```

### Layer 3：回覆真實性評分（Reply Authenticity Score）

**問題：這個回覆是真的有興趣，還是在敷衍？**

```
每則回覆由 A-LI-Auditor 自動評分（滿分 10 分）：

加分項（真實興趣訊號）：
  +3 分：提到自己公司的具體情況（「我們現在確實有這個問題...」）
  +3 分：問了一個有深度的問題（「你們支援哪些 API 供應商？」）
  +2 分：主動提到預算或時間線（「我們下季有評估 AI 工具的計畫」）
  +2 分：提到具體的痛點（「我們的 OpenAI 帳單每月都超標」）
  +2 分：要求更多資訊（「可以寄一個 Demo 連結嗎？」）
  +1 分：回覆時間 < 24 小時（說明有在關注）

減分項（敷衍/假性興趣訊號）：
  -2 分：只有一句話且沒有具體內容（「謝謝，我再看看」）
  -2 分：回覆模板感強（「您好，感謝來訊，目前不需要」）
  -1 分：回覆時間 > 7 天（對話溫度已冷）
  -3 分：要求直接寄報價單（跳過了解需求，通常是在收集比價）

評分對照：
  8-10 分 → 真實 HOT，立刻排 Demo
   5-7 分 → 真實 WARM，繼續對話
   2-4 分 → 低質回覆，降溫，繼續觀察
   ≤ 1 分 → 禮貌性回覆，移至 NURTURE
   
A-LI-Auditor 每週輸出：
  本週 X 則回覆，真實意圖分布：
  HOT 回覆（8+）：Y 則（Z%）
  有效回覆（5-7）：Y 則（Z%）
  低質回覆（≤4）：Y 則（Z%）
  → 若低質回覆 > 50%，代表訊息模板需要重寫
```

### Layer 4：意圖進展追蹤（Intent Progression Tracking）

**問題：這個 Lead 真的在往成交方向走嗎？**

```
意圖進展信號體系（每個 Lead 有動態分數）：

強訊號（+10 分，代表購買意圖明顯提升）：
  ✓ 主動詢問報價或方案
  ✓ 問到可以「下週」或「這個月」試用
  ✓ 要求介紹給其他決策者（「讓我叫 IT 主管一起聊」）
  ✓ 提到競品（「我們在比較你們和另一家...」）

中訊號（+5 分）：
  ✓ 問了 2 次以上的問題
  ✓ 主動分享他們公司的 AI 使用狀況
  ✓ 開了你分享的連結超過 3 分鐘（Smart Links 追蹤）

弱訊號（+2 分）：
  ✓ 回覆了，但只是確認收到
  ✓ 追蹤了公司 LinkedIn 頁面
  ✓ 點了 Email 中的費用試算連結（未完成）

退步訊號（-5 分）：
  ✗ 超過 10 天沒有互動（對話降溫）
  ✗ 說「現在很忙，之後再說」（延後 = 優先級低）
  ✗ 換話題、不正面回應核心問題

A-LI-Auditor 每週審核：
  意圖分數連續 2 週下降的 Lead → 移至 NURTURE，釋放 Kid 精力
  意圖分數連續 2 週上升的 Lead → 升為 HOT，加速跟進
```

### Layer 5：Demo 品質驗證（Demo Quality Gate）

**問題：這場 Demo 是真的有推進，還是只是浪費一小時？**

```
Demo 前查核（A-LI-Auditor 在預約確認後自動觸發）：

  □ 與會者是決策者還是執行者？
      決策者（IT Director / CTO / CEO）→ Green
      執行者（工程師 / 助理）→ Yellow，詢問是否有主管參與
      不確定 → Kid 在確認信中確認
      
  □ 對方確認時有提出任何問題嗎？
      有問具體問題 → 有準備，是真實客戶
      只說「好，那時間見」→ 低確認度，提醒可能 No-Show

Demo 後 24 小時查核（Kid 填入 Asana，Auditor 驗證）：
  必填欄位（缺任何一個 → Auditor 標記為「未完成 Demo 記錄」）：
    □ 與會人數和職稱
    □ 對方提出的 1-3 個具體問題（逐字記錄）
    □ 對方表達的主要顧慮（若有）
    □ 下一步行動（明確日期 + 負責人）
    □ Demo 品質評分（Kid 自評）：1-5 分

  品質評分標準：
    5 分：對方主動問報價，約好下週跟進
    4 分：對方有真實需求，需要評估
    3 分：對方有興趣但無明確時間線
    2 分：對方聽完了，但不置可否
    1 分：對方一開始就說「只是了解一下」

  若 Demo 品質分 ≤ 2 → Auditor 自動通知 Lauren 複盤原因
```

### Layer 6：漏斗完整性稽核（Funnel Integrity Audit）

**問題：Asana 的狀態是真實發生了，還是人工美化了？**

```
A-LI-Auditor 每週五自動執行的 Asana 稽核：

稽核規則 1：無活動即降溫
  Section 03（已觸達）但 > 21 天無任何更新的 Task
  → 自動標記「可能沉默」，通知 Kid 確認狀態
  → 若 Kid 7 天內無回應 → 自動移至 NURTURE

稽核規則 2：跳步驗證
  Task 從「01 Connection Sent」直接跳到「04 Replied HOT」
  中間沒有「02 Connected」和「03 Value Sent」的時間戳記
  → 標記為「跳步異常」，要求 Kid 補充說明

稽核規則 3：時間合理性
  連結請求當天就直接預約 Demo → 異常（太快，可能資料有誤）
  Lead Pool 裡有超過 60 天未觸達的 Lead → 資源浪費，清理或啟動

稽核規則 4：子任務完成率
  每個 Task 應有的子任務（連結請求 / 價值觸達 / 回覆記錄）
  子任務完成率 < 60% 的 Task → 標記為「記錄不完整」

稽核規則 5：Demo 後跟進追蹤
  Demo 完成（Section 07）但 5 天後仍無下一步記錄
  → 標記「跟進缺失」，自動通知 Kid + Lauren

稽核輸出格式（每週五 Asana 自動建立「本週稽核報告」Task）：
  🟢 Green：X 筆 Task 記錄完整，流程正常
  🟡 Yellow：Y 筆 Task 有輕微記錄缺失（清單附上）
  🔴 Red：Z 筆 Task 有嚴重問題（需要 Kid 本週處理）
```

### Layer 7：效果歸因驗證（Attribution Integrity）

**問題：這個成交真的是因為這個渠道 / 這個訊息嗎？**

```
每次成交後，A-LI-Auditor 執行回溯歸因分析：

  Lead 來源：LinkedIn / 104→LinkedIn / Apollo / 轉介
  首次觸達日期 → 成交日期 = 銷售週期長度
  成交前的關鍵互動節點（哪一封訊息讓對方開始認真？）
  訊息模板版本（A 或 B）→ 對成交有貢獻的是哪個版本？

月度歸因報告（Jet + Frank 可查看）：
  ┌────────────────────────────────────────────────┐
  │  本月成交歸因分析                                │
  ├────────────────────────────────────────────────┤
  │  成交 5 件：                                     │
  │  ├ LinkedIn：3 件（平均銷售週期 28 天）            │
  │  ├ 104→LinkedIn：1 件（平均銷售週期 18 天）        │
  │  └ Apollo Email：1 件（平均銷售週期 35 天）        │
  │                                                  │
  │  效果最好的訊息模板：Template-A（IT版）            │
  │  平均回覆率：35% vs Template-B 的 22%             │
  │                                                  │
  │  關鍵成交轉折：第 3 次跟進（Day 7）成功率最高（60%）│
  └────────────────────────────────────────────────┘
```

## A-LI-Auditor：稽核 Agent 完整規格

### 定位與個性

```
A-LI-Auditor 的角色設定：
  - 不相信任何人自填的數字，直到有證據
  - 對「聽起來不錯」的回覆保持懷疑
  - 以「最壞情況假設」做基準，而非最好情況
  - 對 Kid 友善，但對數字絕不手軟
  - 每週固定時間輸出，絕不跳過
  
一句話定位：
  「A-LI-Scout 找人，A-LI-Writer 寫字，
   A-LI-Tracker 記數字，A-LI-Auditor 查數字是不是真的。」
```

### 自動化執行排程

```
每日（自動，不需要 Kid 介入）：
  08:30 — 掃描 Asana：偵測逾期 Task，發出提醒
  18:00 — 掃描當日回覆：計算回覆真實性分數（Layer 3）
           偵測超過 3 次開信但未回覆的 Email Lead（升溫提醒）

每週五 17:00（自動，輸出報告）：
  1. 執行 Layer 6 漏斗完整性稽核
  2. 計算本週各渠道真實有效觸達數（去除沉睡帳號、高退信）
  3. 計算回覆品質分布（HOT / WARM / 低質比例）
  4. 產出「現實調整後」的漏斗數字（vs. 帳面數字）
  5. 建立「本週稽核報告」Task，指派給 Lauren

每月最後一個週五（深度稽核）：
  1. 執行所有 7 層完整審查
  2. 計算 ICP 評分準確率（預測 HOT 實際成交率）
  3. 輸出效果歸因月報（哪個渠道 / 模板 / 訊號真正帶來成交）

```

## 市場意向真實感知機制

這是比數字更重要的問題：**「你怎麼知道這個市場真的有需求？」**

### 三種真實性確認方法

**方法 1：問一個只有真實需求者才能回答的問題**

```
不好的開放問句（任何人都能敷衍回答）：
  「你們有用 AI 嗎？」→ 「有」（沒意義）

好的確認問句（需要真實情境才能回答）：
  「你們現在 OpenAI 和 Azure OpenAI 是分開帳號還是統一？」
  → 如果對方能直接回答，代表他真的在管這件事
  → 如果對方說「我去確認一下」，代表他有在乎但不清楚
  → 如果對方說「這是什麼意思？」，代表他還沒碰到這個問題
  
這三種回答，各自代表不同的成熟度，都是真實資訊。
```

**方法 2：讓 Lead 自己說出痛點**

```
驗證型問句設計原則：
  不要告訴對方他有什麼問題 → 問對方覺得他有什麼問題
  
例子：
  「你們 AI 工具的費用，目前是怎麼追蹤的？」
  
可能的真實回答類型：
  A 型：「我們用 Excel 手動整理，很麻煩」→ 真實痛點，高價值
  B 型：「我們用 [某工具] 在管」→ 已有解決方案，了解競品
  C 型：「IT 部門負責，我不太清楚」→ 非決策者，轉介需求
  D 型：「目前量不大，還好」→ 時機未到，放入 NURTURE

A 型 Lead 的真實感，遠高於你「說服」對方他有問題。
```

**方法 3：Smart Links + 行為追蹤驗證**

```
在價值觸達訊息中放置 Smart Link（費用試算連結、案例 PDF）

行為信號解讀：
  點擊連結 < 30 秒後離開 → 只是隨意點開（低意圖）
  停留 2-3 分鐘        → 有在閱讀（中意圖）
  停留 > 5 分鐘        → 認真在看（高意圖）
  同一 IP 多次訪問      → 在內部討論中（很高意圖）
  分享連結給別人訪問    → 已在內部傳閱（準備購買）

A-LI-Auditor 每日掃描 Smart Links 數據：
  新增高意圖訪問（>3分鐘）→ 立刻通知 Kid，48 小時內跟進
```

## KPI 可信度分級

```
不是所有 KPI 都一樣可信。A-LI-Auditor 對每個指標標注可信度：

⭐⭐⭐⭐⭐ 最高可信度（行為數據，不能造假）：
  連結點擊（Smart Links 追蹤）
  Email 開信 + 點擊（Apollo 追蹤）
  Demo 實際出席（Calendar 確認）

⭐⭐⭐⭐ 高可信度（可觀察，偶有誤判）：
  LinkedIn 連結請求接受率
  Email 退信率
  回覆內容真實性評分（A-LI-Auditor 評）

⭐⭐⭐ 中可信度（自填，有樂觀偏差風險）：
  Lead 分類（HOT / WARM / COLD）
  Demo 品質自評分數
  意圖進展判斷

⭐⭐ 低可信度（容易被美化）：
  「對方有興趣」（需要問清楚具體下一步才算）
  「他說等預算」（可能是委婉拒絕）

⭐ 最低可信度（僅供參考）：
  「感覺市場有需求」（需要數據支撐才成立）
  
每份報告中，A-LI-Auditor 用⭐數量標注每個數字的可信度，
讓 Kid 和 Lauren 知道哪些數字可以決策，哪些需要更多驗證。
```
