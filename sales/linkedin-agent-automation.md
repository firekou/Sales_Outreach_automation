## Part 2：三個 Agent 規格

### 2.1 A-LI-Scout（Lead 偵察 Agent）

**職責邊界**
- 在 LinkedIn Sales Navigator 依 ICP 篩選條件自動搜尋 Lead
- 對每筆 Lead 進行多維度評分（0-100 分）
- 依評分結果分類輸出至 Asana 任務看板
- 每日生成「今日候選 Lead 清單」供 Kid 覆核

**Lead 評分模型**

| 評分項目 | 配分 | 評分邏輯 |
|---------|-----|---------|
| 職稱匹配度 | 30 分 | 決策者 30 / 影響者 15 / 一般員工 5 |
| 公司規模匹配 | 20 分 | 20-100 人 20 / 10-200 人 15 / 邊緣值 5 |
| 產業匹配度 | 20 分 | P1-P2 產業 20 / P3-P4 產業 15 / 其他 5 |
| 活躍度訊號 | 15 分 | 近 30 天發文 / 互動 / 分享 AI 內容 |
| AI 工具痛點訊號 | 15 分 | Profile / 貼文含 OpenAI、API、AI 工具、Token 等關鍵字 |

**Lead 分類標準**

| 分類 | 分數區間 | 行動 |
|-----|---------|-----|
| HOT（熱門）| 70-100 分 | 優先觸達，同日生成草稿 → 審核後 Make.com 自動發送 |
| WARM（潛力）| 50-69 分 | 次日批次觸達，審核後自動發送 |
| COLD（觀望）| <50 分 | 進入培育池，每月觸達一次 |

**觸發條件**
- 每日 07:30 自動執行搜尋（工作日）
- 每週一深度掃描：重新檢視過去 COLD 池，評估是否升級
- Kid 手動新增篩選條件時，即時觸發補充搜尋

**Asana 任務卡輸出格式**

```
任務標題：[HOT] 王小明 ｜ IT 主管 ｜ 睿智科技 ｜ 85分
自定欄位：
  LinkedIn URL：linkedin.com/in/...
  公司官網：...
  公司規模：35 人
  產業：科技軟體
  評分明細：職稱30 / 規模20 / 產業20 / 活躍度10 / AI痛點5
  評分依據摘要：近 14 天分享過 GPT-4 成本管理文章，職稱為資訊技術主管
  推薦訊息模板：Template-B（IT 主管版）
  建立日期：2026-05-28
  狀態：待 Kid 覆核
```

**人類覆核點**
- Kid 需在收到 HOT Lead 的當日工作時段內確認：通過觸達 / 退回重評 / 手動修改分類
- 每週一 Lauren 抽查 10% 的 REJECT 清單，確認 Agent 沒有誤殺優質 Lead

### 2.2 A-LI-Writer（訊息撰寫 Agent）

**職責邊界**
- 依 Lead 的 Scout 資料，自動生成高度個人化的訊息草稿
- 從已核准模板庫中選取最適合的基底模板
- 填入個人化變數（姓名、公司、職稱、近期活動、痛點訊號）
- 輸出草稿，等待 Kid 核准後才送出

**訊息設計原則**
- 第一觸（Connection Request）：≤ 300 字元，不直接推銷，建立連結感
- Follow-up 1（D+3，對方接受連結後）：≤ 500 字，點出痛點，提供具體價值
- Follow-up 2（D+10，未回覆則發送）：≤ 300 字，輕量再觸達，附案例或數據
- Follow-up 2 後仍無回應 → 進入 30 天冷卻期，不再主動觸達

**模板庫分類**

| 模板代碼 | 適用對象 | 核心訊息主軸 |
|---------|---------|------------|
| Template-A | CTO / 技術主管 | 多供應商 API 管理成本可視化 |
| Template-B | IT 主管 / 資訊長 | 部門 Token 預算管控、避免超標 |
| Template-C | 採購 / 行政主管 | 統一發票、台幣計費、節省採購複雜度 |
| Template-D | 數位轉型主管 | AI 投資 ROI 可量化呈現 |
| Template-E | 行銷 / 顧問公司 | 多客戶 AI 工具費用分攤管理 |

**Asana 子任務輸出格式**

```
子任務標題：[草稿待審] 第一觸訊息 ｜ 王小明 ｜ 2026-05-28
附件內容：
  使用模板：Template-B（IT 主管版）
  個人化變數填入：
    姓名：王小明
    公司：睿智科技
    近期活動：您上週分享了關於 GPT-4 API 費用控制的文章，讓我印象深刻
    痛點對應：多供應商管理、月底超標
  訊息草稿全文：「王主管您好，...（300字以內完整草稿）...」
  建議發送時間：週二 / 週四上午 10:00-11:00
  狀態：等待 Kid 核准
```

**人類覆核點**
- Kid 必須逐筆核准或退回每則草稿，Agent **不得**自行發送任何訊息
- 若 Kid 有修改，修改版本須回存至 Asana 作為版本記錄
- 每月月底 Lauren 審核本月送出訊息樣本（抽 20%）

### 2.3 A-LI-Tracker（追蹤分析 Agent）

**職責邊界**
- 監控所有觸達 Lead 的狀態變化（連結接受、訊息已讀、已回覆、會議排程）
- 自動計算各階段轉換率並更新 Asana 看板
- 每日生成「昨日觸達成效摘要」
- 每週生成「本週業績漏斗報告」
- HOT Lead 在 Follow-up 2 後仍無回應 → 自動提醒 Kid 判斷是否人工介入
- 偵測異常訊號（回覆率驟降、特定模板失效）→ 自動通知 Lauren

**觸發條件**
- 每日 18:00 自動拉取當日數據並更新看板
- 每週五 16:00 自動生成週報，推送至 Asana + 通知 Kid / Lauren
- 轉換率連續 3 天低於警戒線 → 即時通知 Lauren

**週報輸出格式**

```
本週 LinkedIn 觸達成效週報（2026-W22）
生成時間：2026-05-29 16:00
────────────────────────────
漏斗數據：
  本週新增 Lead：42 筆
    HOT：12｜WARM：18｜COLD：8｜REJECT：4
  已發出連結請求：28 筆
  連結接受率：64.3%（18/28）
  已回覆（任何形式）：6 筆
  回覆率：33.3%（6/18 已接受連結者）
  進入商務討論：2 筆
  本週新增潛在機會金額：NT$ 約 198,000

模板效能排行：
  1. Template-B：回覆率 45%（最佳）
  2. Template-D：回覆率 38%
  3. Template-A：回覆率 22%（需檢討）

警示事項：
  Template-A 本週回覆率低於歷史均值 15 個百分點，建議安排 A/B 測試
```

## Part 5：月度迭代機制

### 5.1 訊息模板 A/B 測試機制

**測試週期：** 每 2 週一個循環

```
Step 1 — 觸發（每月第一個週一）
  A-LI-Tracker 生成上月各模板效能報告
  回覆率最低的模板自動標記「待優化」

Step 2 — 提案（Lauren 在 3 個工作日內）
  提出 1-2 個改版假設
  例：「Template-A 回覆率低，假設是開頭太直白，
       改版 A-v2 改為先提對方近期貼文再切入」
  草稿提交 Kid 與 Jet 確認

Step 3 — 分組（A-LI-Writer 執行）
  符合條件的新 Lead 隨機分 A / B 組（各 50%）
  A 組用現行版本，B 組用改版模板
  測試期間不混用

Step 4 — 觀察（2 週）
  A-LI-Tracker 分別追蹤 A / B 組的接受率與回覆率
  每週五週報中標示 A/B 測試中間數據

Step 5 — 決策（測試期結束後週一，Lauren 主持）
  B 組回覆率比 A 組高 ≥ 10 個百分點 → B 升為正式模板
  差異 < 10 個百分點 → 維持 A 組，記錄假設不成立原因
  B 組更差 → A 組繼續，記入改版假設錯誤清單

Step 6 — 版本封存
  每個模板版本存入 Asana 模板庫（版本號、測試結果、退役原因）
  Kid 於月度回顧分享質性洞察，補充定量數據
```

**決策人：** Lauren（主持）× Kid（質性確認）× Jet（若涉及商務表達則知會）

### 5.2 ICP 定義季度校正

**校正頻率：** 每季最後月份第三週（3月、6月、9月、12月）

**提前觸發指標**
- 連續 2 個月 ICP 命中率低於 50%
- 成交客戶與 ICP 定義出現系統性偏差
- Alice 回報產品功能上線後適用客群有調整

**校正流程**

```
準備階段（校正前 1 週，Lauren 召集）：
  - A-LI-Tracker 匯出本季 Lead 資料：職稱/產業/規模分佈、成交 vs 未成交對比
  - Kid 提供「本季最容易成交的前 10 個客戶畫像描述」
  - Alice 提供「產品最適合解決的痛點清單」更新

校正會議（Lauren 主持，Alice + Kid + Jet 參與）：
  - 對比 Scout 數據與實際成交客戶的 Profile 差異
  - 決定是否調整 ICP 篩選條件（職稱 / 產業 / 規模區間）
  - 決定是否調整 Lead 評分模型各項配分

執行：
  - 更新後 ICP 定義版本號 +1，存檔記錄
  - 若調整幅度超過 3 個維度同時變動，需 Jet 連署確認
  - 若涉及目標市場轉移，需 Frank 拍板

生效後追蹤：
  - 調整後 4 週，Lauren 確認新版 ICP 命中率是否改善
  - 若無改善，重新召集討論
```

### 5.3 Agent Prompt 版本管理

**觸發條件**
- 訊息品質達標率連續 2 個月低於 90%
- Kid 回報草稿「感覺不像人說的話」超過 3 次 / 月
- 平台政策變動（LinkedIn 調整字數限制或演算法）
- 新產品功能上線，現有 Prompt 未納入最新賣點

**版本命名規則**
```
格式：[Agent 代碼]-[功能模組]-[版本號]
範例：A-LI-Writer-MSG-v1.2（訊息生成模組第 1.2 版）
```

**審核流程**

| 修改幅度 | 核准人 | SLA |
|---------|--------|-----|
| 小幅調整（單一詞彙 / 句子）| Kid 提案，Lauren 核准 | 1 個工作日 |
| 中幅修改（段落重構 / 策略調整）| Lauren 提案，Jet 知會 | 3 個工作日 |
| 大幅重寫（整體框架改版）| Lauren 提案，Jet 連署，Frank 知會 | 5 個工作日 |

**版本控制原則**
- 所有 Prompt 版本存放於 Asana 文件庫（指定資料夾）
- 每次修改需記錄：修改人 / 日期 / 原因 / 改動內容摘要
- 正式版本需 Lauren 核准後才能部署至 Agent
- 舊版本不刪除，保留作為 rollback 依據

## 附錄：Lauren 的重要提醒

**關於平台上限**
LinkedIn Sales Navigator 每日連結請求約 20-25 則上限，超過會被限流甚至帳號警告。Agent 設計為每日 8-15 筆觸達，留有安全緩衝。Sales Navigator 帳號費用走 L2，Lauren 可核准。

**關於人工核准的不可妥協性**
訊息草稿「Kid 必須逐筆核准」這一點不能妥協。台灣個資法和 LinkedIn 使用條款都要求自然人對發出的訊息負責，自動大量發送有平台封號風險。讓 Kid 在這個環節保持人工是保護公司，也保護 Kid。

**關於系統的真正價值**
Agent 做好了，Kid 的時間就能從「找 Lead、打草稿」移轉到「真正跟對的人說話」。Kid 每天 1.5 小時覆核做好，剩下的 6.5 小時業務時間應該全部放在把已進入商務討論的 Lead 推進到成交。這才是這套系統的真正價值。
