## 為什麼需要這份文件

S2 的整條流程（抓職缺 → 找決策者 → 發訊息）技術上都能跑，但**最容易崩掉的是「我們憑什麼聯繫對方」這個合理性**。

如果第一封訊息讓對方覺得「你只是拿我的徵才當藉口來推銷」，那麼前面所有的抓取、評分、自動化都白做了。

**本文件定義：從「徵才訊號」到「合理接觸」之間，那座必須存在的邏輯橋。**

---

## 一、第一性原理

> **徵才行為是一個「狀態證據」，不是「採購訊號」本身。**

職缺證明了公司處於某個**狀態**（正在投資 AI、正在規模化開發），我們不是因為他徵才而推銷，而是從那個狀態**推論出一個下游的必然需求**——那個需求才是我們存在的合理理由。

### 合理性的定義

> **合理性 = 推論鏈中間那一步，對方自己是否會點頭。**

| | 壞的推論（藉口感） | 好的推論（合理） |
|---|---|---|
| 結構 | 你徵 AI 人才 → 所以買我們工具 | 你徵第 N 個 AI 人 → **代表開發已從實驗走向規模化** → 規模化必然遇到多人協作下的 API 用量失控 → 這正是我們解的 |
| 差別 | 跳一步，靠斷言 | 中間有一句**對方會對自己點頭**的話 |
| 對方感受 | 「你只是想賣東西」 | 「你好像真的懂我現在的處境」 |

### 互補定位原則

讓合理性站穩的關鍵框架：**互補，而非取代或無關。**

> 「我們不是要取代你要徵的人——你徵的這個人到職後，第一個會撞到的問題就是我們解決的。」

這一句把「徵才」和「我們的工具」從**不相干**變成**互補**，合理性就立住了。

---

## 二、合理性閘門（Legitimacy Gate）

在生成任何話術前，必須通過三道檢查，否則 S2 不該觸發、或降級為觀察池：

| 閘門 | 檢查問題 | 不通過的處理 |
|------|---------|------------|
| **Gate 1：狀態證據是否充分** | 訊號足夠證明「規模化／投資」這個狀態嗎？單一一則模糊職缺 ≠ 規模化 | 單則模糊職缺 → 降為 C 級觀察池，不打 |
| **Gate 2：中間步驟是否為真** | 我們要接觸的這個人，他的角色是否真的會碰到那個下游問題？ | 角色對不上 → 換對象，或不打 |
| **Gate 3：互補性是否成立** | 我們和他要徵的人，是互補還是無關？ | 無關 → 不打 |

> **實作銜接：** 這一層補在 `02-S2-104-intent-automation-spec.md` 的 Step 6（關鍵字過濾）之後、Step 7（Apollo 查詢）之前。現有設計只做關鍵字評分，沒有驗證邏輯鏈是否成立。

## 三、三種對象的客製化接觸理由

核心：**每種對象的「中間步驟」不一樣，因為那句話必須對「他的角色」為真。**

---

### 對象 1｜AI 開發／流程部門（建造者 Builders）

| 項目 | 內容 |
|------|------|
| **如何識別** | 職缺＝AI 流程自動化／RPA／AI 應用開發／ML 工程師，**且部門已有人**（不是從零招第一個） |
| **徵才證明的狀態** | 他們正在 build，且從 PoC 走向 production／規模化 |
| **中間步驟（他會點頭的話）** | 「多個 AI 流程同時上線、多個工程師同時開發，API 用量會變得難追蹤、難控制」 |
| **接觸理由的本質** | **相鄰專業的問題預警**——不是賣東西給你要徵的人，而是看到你們在規模化，而規模化的下一關就是用量治理 |
| **第一訊息骨架** | ① 觀察：你們在擴編 AI 流程團隊<br>② 推論：多人＋多流程＝用量分散難管<br>③ 價值：在團隊變大前先管好<br>④ 互補：你徵的人到職就直接受益，不用自己造輪子 |
| **要避免** | 不要假設他們已經很痛（可能才剛建）。用「**在問題變大之前**」的預防框架 |
| **語氣定位** | 同行之間的經驗提醒，平視、不居高臨下 |

---

### 對象 2｜技術人員（使用者／影響者 Practitioners）

| 項目 | 內容 |
|------|------|
| **如何識別** | Apollo 找到的工程師、或職缺團隊裡現有技術人員。**非決策者** |
| **徵才證明的狀態** | 這個技術棧是真的在用（徵同類人＝持續投入） |
| **中間步驟（他會點頭的話）** | 「管多個 LLM 的 key、追蹤誰燒了多少 token、避免某個實驗把預算燒光——這些雜事很煩」 |
| **接觸理由的本質** | **工程師對工程師的工具分享**——合理性來自技術同儕善意，不是商業關係 |
| **第一訊息骨架** | ① 身分：同樣在做 LLM 應用的人<br>② 具體技術痛點（只有真做過的人講得對）：key 管理／用量歸因／多模型切換<br>③ 工具分享：免費試、自己玩<br>④ 不談合約、不談決策 |
| **要避免** | 對工程師**不要用商業語言**（ROI／方案／報價）。目標是把他變成 internal champion，不是當天成交 |
| **語氣定位** | 技術同好分享，輕鬆、具體、不帶銷售壓力 |

---

### 對象 3｜AI 管理者／決策者（治理者 Governors）

| 項目 | 內容 |
|------|------|
| **如何識別** | CTO／數位長／Head of AI／技術 VP，透過 Apollo 在徵才公司中定位 |
| **徵才證明的狀態** | **徵才本身就是投資的證據**——他批准了 AI 職缺＝他有 mandate、有預算、有 KPI |
| **中間步驟（他會點頭的話）** | 「我投資建 AI 團隊，但這些 AI 工具的錢花得值不值？多團隊用不同工具，我怎麼治理、怎麼控成本？」 |
| **接觸理由的本質** | **治理與投資回報的對話**——你正在投錢建能力（徵才為證），投資的另一面就是治理：成本可見性、用量管控、多工具整合 |
| **第一訊息骨架** | ① 觀察：你們在投資建立 AI 團隊（以徵才為據）<br>② 戰略提問：投資的同時，工具成本治理有方案了嗎？<br>③ 價值：我們給管理者要的那層可見性與控制<br>④ 高度：不講功能，講治理與 ROI |
| **要避免** | 不要陷技術細節、不要像在賣給工程師。站在「**你作為管理者會擔心的事**」這個高度 |
| **語氣定位** | 對等的策略對話，簡潔、有高度、尊重對方時間 |

## 四、路由邏輯：哪種職缺 → 接觸哪種人 → 用哪套理由

| 職缺類型 | 證明的狀態 | 首選接觸對象 | 用哪套理由 |
|---------|-----------|------------|-----------|
| 多則 ML／AI 工程師 | 規模化開發 | 技術 lead／CTO | 建造者 or 治理者 |
| RPA／流程自動化 | 流程 AI 化啟動 | 流程部門主管 | 建造者 |
| Prompt Engineer | 正在部署 LLM 應用 | 技術 lead | 建造者＋技術人員 |
| AI 產品經理 | AI 產品化 | 產品／決策者 | 治理者（談 AI 單位成本） |
| 數位轉型專員 | 組織級 mandate | 數位長／高層 | 治理者 |

### 多線觸及原則

同一家公司若同時找到工程師＋CTO，可以**用不同理由各打一次**（工程師走技術分享、CTO 走治理）——但要節流：

- 同一家公司**同一週內最多接觸 2 個不同對象**
- 兩個對象的訊息**理由必須不同**（不可同一套話術改名字）
- 在 Asana 同公司 Task 互相標注，避免 Kid 重複跟進造成轟炸感

## 五、合理性自檢清單（話術上線前）

每一條 S2 話術（無論 Claude 生成或 Kid 手寫），上線前用這份清單自檢：

- [ ] 有沒有出現「正好」「剛好」這類**巧合詞**？（有 → 重寫，這是藉口的破綻）
- [ ] 訊息裡有沒有一句是「**對方會對自己點頭**」的觀察？（沒有 → 合理性不足）
- [ ] 我有沒有把「他要徵的人」和「我們的工具」定位成**互補**？
- [ ] 這封訊息換成任何一家公司都能用嗎？（能 → 太通用，不夠個人化）
- [ ] 對象角色和理由對得上嗎？（對工程師談 ROI、對 CTO 談 key 管理 → 錯位）
- [ ] 第一封訊息有沒有**先講下游問題、後講產品**？（先講產品 → 推銷感）

---

## 六、總結

S2 接觸合理性的核心，一句話：

> **我們不是因為你徵才而出現，而是因為你徵才所「證明的狀態」會帶來一個你那個角色一定會撞到的問題——而我們剛好站在那個問題前面。**

三種對象，三條不同的邏輯橋，但共用同一個原理：**找到對方角色裡那個會點頭的中間步驟。**
